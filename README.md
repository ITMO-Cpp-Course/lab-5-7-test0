Проект по разработке поискового индекса для текстовых документов.  
Проект выполняется в рамках **трёх лабораторных работ** и последовательно развивается от простого in-memory индекса до полноценного CLI-приложения для индексирования файлов на диске.

---

## Лабораторная работа №5
### Часть 1 — In-memory индекс документов

### Введение

Во многих прикладных задачах требуется быстро искать информацию в большом количестве текстовых файлов:

- логи
- документация
- заметки
- исходный код

В основе системы лежит **инвертированный индекс**:

- каждый документ имеет уникальный идентификатор;
- текст документа разбивается на слова;
- по слову можно быстро получить список документов, в которых оно встречается.

В данной части реализуется **ядро поискового индекса**, полностью работающее в памяти и не зависящее от файловой системы.

---

### Цель работы

- Реализовать структуру данных для поиска слов по документам
- Научиться передавать и хранить объекты без лишних копирований
- Освоить базовое проектирование прикладного API

---

### Основные сущности

#### `Document`
Представляет один текстовый документ.

Документ:
- имеет идентификатор;
- имеет имя;
- содержит текст;
- может передаваться в индекс и храниться в нём.

#### `InvertedIndex`
Основная структура данных индекса.

Позволяет:
- добавлять документы;
- удалять документы;
- искать документы, содержащие заданное слово;
- получать количество вхождений слова в каждом документе.

#### `DocumentBuilder`
Вспомогательный объект для подготовки документов.

Позволяет:
- создавать `Document` из имени и текста;
- разбивать текст на слова;
- приводить слова к единому виду (например, `lowercase`).

---

### Тесты

Тестами должна быть покрыта вся логика, включая:
- добавление и удаление документов;
- поиск слов;
- корректность подсчёта вхождений.

---

### Результат

- In-memory индекс документов
- Понятный и расширяемый API для поиска документов по словам

## Лабораторная работа №6
### Часть 2 — Обработка ошибок и транзакционные обновления

### Введение

В реальных программах операции с данными могут завершаться неудачно:

- неверный ввод;
- логические ошибки;
- ошибки при обновлении состояния.

Важно, чтобы при ошибках:
- состояние системы оставалось корректным;
- частично выполненные изменения не сохранялись.

В этой части добавляется **безопасный слой управления индексом** с использованием транзакций и `std::expected`.

---

### Цель работы

- Научиться возвращать ошибки через `std::expected`
- Реализовать атомарные обновления состояния
- Использовать RAII для автоматического отката изменений

---

### Основные сущности

#### `IndexStore`
Объект, управляющий индексом и документами.

Позволяет:
- добавлять документы;
- удалять документы;
- выполнять поиск;
- начинать транзакцию обновления.

Все операции возвращают `std::expected`.

#### `UpdateTransaction`
RAII-объект для обновления `IndexStore`.

Позволяет:
- выполнять изменения над индексом;
- зафиксировать изменения вызовом `commit()`;
- автоматически отменить изменения, если `commit()` не был вызван.

#### `Result`
Тип-обёртка на базе `std::expected`, используемый всеми публичными методами API.

---

### Поведение при ошибках

**Требование:**  
если операция завершается ошибкой, состояние `IndexStore` **не должно измениться**.

---

### Тесты

Тестами должна быть покрыта логика:
- корректной обработки ошибок;
- отката изменений при неудачной транзакции;
- успешного применения изменений.

---

### Результат

- Безопасный слой управления индексом
- Транзакционный механизм обновлений
- API с явной и строгой обработкой ошибок

## Лабораторная работа №7
### Часть 3 — Работа с файлами и CLI

### Введение

На этом этапе индексатор начинает работать с реальными данными на диске.

Типичный сценарий:
- есть директория с текстовыми файлами;
- каждый файл — отдельный документ;
- по содержимому файлов строится индекс;
- пользователь выполняет поиск через консоль.

Подобные инструменты применяются для анализа логов, документации и больших наборов текстов.

---

### Цель работы

- Научиться читать файлы с диска
- Использовать `std::filesystem` для обхода директорий
- Реализовать консольный интерфейс (CLI)
- Подготовить архитектуру для возможной многопоточности

---

### Основные сущности

#### `FileLoader`
Компонент для работы с файловой системой.

Позволяет:
- обходить директории;
- находить текстовые файлы;
- загружать содержимое файлов в память.

#### `IndexApplication`
Высокоуровневый компонент приложения.

Позволяет:
- построить индекс по директории;
- выполнять поиск по слову;
- получать базовую статистику по индексу.

#### `CLI`
Консольный интерактивный интерфейс приложения.

Поддерживаемые команды:
- `build` — построить индекс по директории;
- `query` — выполнить поиск слова;
- `stats` — вывести статистику по индексу.

---

### Условие под многопоточность

Архитектура приложения должна позволять:
- распараллеливание чтения файлов;
- распараллеливание подготовки документов.

> Реализация многопоточности **не обязательна**, важна архитектурная готовность.

---

### Тесты

Тестами должна быть покрыта логика:
- загрузки файлов;
- построения индекса;
- работы CLI-логики.

---

### Итог проекта

После выполнения всех лабораторных работ проект представляет собой:

- прикладной инструмент для индексирования текстовых файлов;
- расширяемую и чистую архитектуру;
- единый проект из трёх логически связанных частей.
